# Review Blog Post Skill

这个 skill 用于检查和修正中文博客文章的语句、格式、标点和一致性问题。

## 适用范围

- 中文博客文章
- 技术类博客内容
- 中英文混合的文章

## 检查和修正项目

### 1. 语句通顺性

#### 常见问题及修正方法

**词语搭配不当**
- `贡献的` → `提供的`（best practice 用"提供"更恰当）
- `工作量占大头的工作` → `工作中占大头的工作`（避免重复"工作"）
- `由...转而` → `从...转而`（正确的搭配）
- `持续的维护负责` → `持续维护和负责`（添加连接词）
- `好减少` → `以减少`（更正式的表达）

**语句冗长或结构不清**
- 将长句拆分为多个短句，使逻辑更清晰
- 去掉不必要的赘词（如"别的"、"这样子的"）
- 使用标点分隔，增强可读性

**示例：**
```
修正前：如果产品团队不开始使用的话，我是永远不知道这样的诉求的
修正后：如果产品团队不开始使用的话，我永远不会知道这样的诉求
```

**句子结构优化**
- `就...来看` → `从...来看`（更自然的表达）
- `对于...，我对...` → `对于...，我就...`（避免重复主语）
- 适当使用破折号（——）连接相关内容

**示例：**
```
修正前：正如我在[《回归文本》](/posts/file-system/)里谈到的一样。

而现在的软件开发...
修正后：正如我在[《回归文本》](/posts/file-system/)里谈到的，现在的软件开发...
```

### 2. 大小写一致性

#### 英文缩写和专有名词

必须使用正确的大写：
- `ai` → `AI`
- `llm` → `LLM`
- `pm` → `PM`
- `pr` → `PR`
- `tpm` → `TPM`
- `prd` → `PRD`

#### 产品名称和工具

保持官方大小写：
- `claude code` → `Claude Code`
- `chatGPT`/`chatGPT2` → `ChatGPT`
- `agentic ai tool` → `Agentic AI Tool`

#### 技术术语

保持标准大写：
- `nl2sql` → `NL2SQL`
- `ai-driven` → `AI-driven`（连字符形式）
- `workflow` → `workflow`（全小写）

#### 通用名词

作为通用名词时保持小写：
- `agent`（不是专有名词时小写）
- `tool`
- `skill`

**注意：** 文章中 `Agent` 作为章节标题时可大写，但在正文中作为通用概念时应小写为 `agent`。

### 3. 标点符号规范

#### 中文标点

- 使用中文逗号：`，`
- 使用中文句号：`。`
- 使用中文问号：`？`
- 使用中文感叹号：`！`
- 使用中文顿号：`、`
- 使用中文冒号：`：`
- 使用中文引号：`""`
- 使用破折号：`——`（长破折号）

#### 英文标点

在以下情况使用英文标点：
- 代码引用：使用反引号 `` ` ``
- 百分比：`50%-60%`（使用英文连字符）
- 英文缩写：`if necessary`（保持英文逗号）

#### 顿号与逗号

- 并列名词短语：使用顿号 `、`
  - 示例：`部署、监控、测试`
- 并列句子或较长短语：使用逗号 `，`
  - 示例：`开发不再处理简单的事务，开发甚至不再敲打命令行`

#### 句号与分号

- 完整句子：使用句号 `。`
- 分段落的独立观点：可以分成多句，每句用句号

### 4. 空格规范

#### 中英文之间

**必须添加空格的情况：**
- 中文和英文单词之间：`使用 Claude Code 工具`
- 中文和数字之间：`3-8 分`、`100 年`
- 中文和英文标点之间（特殊情况）

**示例：**
```
修正前：利用ChatGPT做NL2SQL
修正后：利用 ChatGPT 做 NL2SQL
```

#### 英文词组内部

- 保持英文单词间的单个空格
- 不要在连字符两侧添加空格：`AI-driven`（正确）vs `AI - driven`（错误）

#### 特殊情况

- 代码块、行内代码：按原样保留
- 链接文本：`[《回归文本》](/posts/file-system/)`（中文标题不加空格）
- 引用块：`> 文本`（`>` 后有一个空格）

### 5. 副词"的/地/得"的用法

- 形容词修饰名词：用"的"
  - 示例：`悲观的看法`、`新的变化`

- 副词修饰动词：用"地"
  - 示例：`确切地说`、`有效地使用`、`自然而然地`

- 补语：用"得"
  - 示例：`说得很清楚`、`做得很好`

**常见错误：**
```
错误：确切的说（应该用"地"）
正确：确切地说

错误：有效的使用（应该用"地"）
正确：有效地使用
```

### 6. 常见口语化表达的规范化

- `这样子的` → `这样一个`或`这样的`
- `回到...之上` → `回归到...之上`
- 去掉句末的"的"：`我是永远不知道这样的诉求的` → `我永远不会知道这样的诉求`

## 修正流程

### 步骤 1：通读全文

- 快速浏览，理解文章主旨
- 标记明显的语句不通顺处
- 注意大小写不一致的地方

### 步骤 2：逐段检查

对每一段进行以下检查：

1. **语句通顺性**
   - 读起来是否流畅
   - 有无词语搭配不当
   - 句子结构是否清晰

2. **大小写检查**
   - 扫描所有英文单词
   - 检查专有名词大小写
   - 确认技术术语拼写

3. **标点符号**
   - 中英文标点是否混用
   - 逗号、句号、顿号使用是否正确
   - 是否需要分句

4. **空格规范**
   - 中英文之间是否有空格
   - 数字和中文之间是否有空格
   - 连字符使用是否正确

### 步骤 3：系统性修正

使用 Edit 工具进行修改：

```bash
# 示例：批量修正大小写
sed -i '' 's/\bai\b/AI/g' file.md
sed -i '' 's/\bllm\b/LLM/g' file.md

# 添加空格（需谨慎，建议手动）
# 中文与英文之间
sed -i '' 's/\([一-龥]\)\([a-zA-Z]\)/\1 \2/g' file.md
sed -i '' 's/\([a-zA-Z]\)\([一-龥]\)/\1 \2/g' file.md
```

**注意：** 批量替换可能影响代码块和链接，建议使用 Edit 工具逐个修改关键位置。

### 步骤 4：验证修改

- 重新通读修改后的段落
- 确认没有引入新的错误
- 检查修改是否保留了原意

## 常见修正示例

### 示例 1：语句优化

```markdown
修正前：
闲话说的有点多，回到开发流程的事情上，就我们最近对ai的能力探索下，我们发现每个角色的边界正在发生变化，对于这个变化，我的看法是很悲观的，确切的说，开发人员的空间正在变小。

修正后：
闲话说的有点多，回到开发流程的事情上。从我们最近对 AI 的能力探索来看，我们发现每个角色的边界正在发生变化。对于这个变化，我的看法是很悲观的，确切地说，开发人员的空间正在变小。
```

**修正点：**
- 分成多个短句，增强可读性
- `就...来看` → `从...来看`
- `ai` → `AI`（大写）
- 添加中英文空格
- `确切的说` → `确切地说`（副词用"地"）

### 示例 2：大小写和空格

```markdown
修正前：
我们团队从利用chatGPT2做nl2sql以来，一直追逐着所有AI的可能性。

修正后：
我们团队从利用 ChatGPT 做 NL2SQL 以来，一直追逐着所有 AI 的可能性。
```

**修正点：**
- `chatGPT2` → `ChatGPT`
- `nl2sql` → `NL2SQL`
- 添加中英文之间的空格

### 示例 3：词语搭配

```markdown
修正前：
产品的角色由精心负责每一项工作转而开始维护文档。

修正后：
产品的角色从精心处理每一项工作转而开始维护文档。
```

**修正点：**
- `由...转而` → `从...转而`（正确搭配）
- `负责` → `处理`（更准确的动词）

### 示例 4：标点和结构

```markdown
修正前：
正如我在[《回归文本》](/posts/file-system/)里谈到的一样。

而现在的软件开发3-8分的开发工作，无非就是在此基础上，嵌入新的一些变化，需求变化。

修正后：
正如我在[《回归文本》](/posts/file-system/)里谈到的，现在的软件开发 3-8 分的开发工作，无非就是在此基础上嵌入新的变化——需求变化。
```

**修正点：**
- 合并两段，使用逗号连接
- 去掉"一样"（赘词）
- 使用破折号（——）强调说明
- 添加数字和中文之间的空格

## 检查清单

在完成修正后，使用以下清单进行最终检查：

- [ ] 所有句子读起来流畅自然
- [ ] AI、LLM、PM 等缩写全部大写
- [ ] Claude Code、ChatGPT 等产品名称大小写正确
- [ ] 所有中英文之间都有空格
- [ ] 数字和中文之间有空格
- [ ] 中文标点符号使用正确
- [ ] 顿号和逗号使用恰当
- [ ] "的/地/得"使用正确
- [ ] 没有"这样子的"等口语化表达
- [ ] 词语搭配恰当（如"从...转而"、"确切地说"）
- [ ] 长句已拆分，逻辑清晰
- [ ] 去掉了不必要的赘词

## 使用工具建议

### 推荐工具

1. **Edit 工具**：逐个修改具体问题
2. **Bash + sed**：批量修正大小写（需谨慎）
3. **Read 工具**：反复阅读验证修改

### 避免的做法

- 不要盲目批量替换，可能影响代码块和链接
- 不要过度修改，保留作者的写作风格
- 不要改变原意，只修正语句和格式

## 注意事项

1. **保留作者风格**：只修正明显错误，不改变作者的表达习惯
2. **代码块保护**：不要修改代码块、行内代码和链接中的内容
3. **渐进式修改**：先修正大问题，再调整细节
4. **验证原意**：每次修改后确认没有改变原意
5. **创建备份**：修改前可以创建备份文件

## 总结

这个 skill 的目标是让中文技术博客更加规范、专业和易读。通过系统化的检查和修正，可以显著提升文章质量，同时保持作者的原始意图和风格。
